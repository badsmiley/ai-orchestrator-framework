# Operating Model — Модель работы

## Два уровня (минимум)

### Уровень 0: Оркестратор (Orchestrator)

**Что это:** Главный AI-агент, который взаимодействует с человеком (или внешним миром).

**Зона ответственности:**
- Получать цели и приоритеты от человека/внешнего источника
- Декомпозировать большие задачи на подзадачи
- Спавнить и управлять субагентами (департаментами)
- Держать глобальную картину: что активно, что заблокировано, что дальше
- Отправлять **micro-reports** после каждого значимого шага

**Принципы:**
- Не задавать лишних вопросов — сначала проверить контекст, файлы, память
- Не отправлять поломанные сообщения во внешние каналы
- Перед внешними действиями (email, посты) — спрашивать подтверждение
- Писать в файлы то, что нужно запомнить

---

### Уровень 1: Департаменты (Departments)

**Что это:** Субагенты, каждый отвечает за отдельный проект, домен или функцию.

**Примеры:**
- `LCH Dev` → LithuanianCoursesHub (курсы литовского)
- `HELIX Web` → Helix Novus Org (образовательные услуги)
- `Ops/Infra` → Инфраструктура, деплой, CI/CD
- `Research/SEO` → Исследования, контент

**Зона ответственности:**
- Выполнять задачи в рамках своего проекта/домена
- Не писать человеку напрямую — репортовать Оркестратору (L0)
- Держать свой статус в актуальном состоянии
- Документировать решения и уроки

**Принципы:**
- Micro-reports лучше молчания
- Если заблокирован — сразу сообщить Оркестратору
- Каждый департамент имеет свой repo/workdir и session

---

## Коммуникация

### Micro-report формат

Каждый департамент отправляет Оркестратору:

```markdown
**{PROJECT} — {STATUS}**

✓ Что сделано: 1-2 предложения
→ Что дальше: следующий шаг
⚠ Блокировки: (если есть)
```

### Пример

```
**LCH Dev — В процессе**

✓ Stripe интеграция добавлена в env vars
→ Следующий: webhook для payment events
⚠ Жду от Олега: STRIPE_WEBHOOK_SECRET
```

---

## Параллелизм

Все департаменты (L1) работают **параллельно**, как команда. Нет единого приоритета — каждый движется по своей задаче, Оркестратор просто координирует.

---

## Глоссарий

| Термин | Определение |
|--------|-------------|
| Orchestrator | Главный агент (L0), управляет системой |
| Department | Субагент (L1), отвечает за проект/домен |
| Worker | Исполнитель (L2), опционально |
| Micro-report | Краткий отчёт от департамента к оркестратору |
| Session | Изолированный контекст агента |
| Workdir | Рабочая директория проекта |
